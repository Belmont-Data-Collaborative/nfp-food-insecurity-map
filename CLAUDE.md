# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

**Static Leaflet frontend + Python data pipeline**, end-to-end. The pipeline ingests Census ACS, CDC PLACES, USDA LILA, NFP partners, and Giving Matters (CFMT) nonprofit data from S3, normalizes it (GEOID zero-padding, MSA filter, 2010→2020 tract crosswalk), and emits flat `*.geojson` + `*.csv` + `config.json` files into `data/`. The three HTML pages ([index.html](index.html), [Map.html](Map.html), [About.html](About.html)) are pure static and read straight from `data/` via `fetch()` — no backend, no build step.

This directory is the merge of two precursors (kept elsewhere for reference only): the Streamlit app in `../dropbuild-nfp-food-insecurity-map-main/` supplied the pipeline, and the Leaflet design mockup in `../nfp-map-claude-design/` supplied the UI.

## Commands

```bash
# One-time setup
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env                 # keep USE_MOCK_DATA=true for local dev

# Seed mock inputs so the pipeline has something to process locally
python scripts/generate_mock_data.py
python scripts/generate_mock_parquet.py

# Run the pipeline end-to-end — populates data/ with everything the frontend needs
python -m pipeline

# Individual pipeline steps
python -m pipeline --step geo             # TIGER boundaries (tracts, ZCTAs, counties, MSA)
python -m pipeline --step census_acs      # ACS income/poverty/population
python -m pipeline --step health_lila     # CDC PLACES diabetes/hypertension/obesity
python -m pipeline --step usda_lila       # USDA LILA with 2010→2020 crosswalk
python -m pipeline --step giving_matters  # CFMT nonprofits (graceful skip if disabled)
python -m pipeline --step partners        # NFP partners (Nominatim geocoding)
python -m pipeline --step export          # Refresh data/config.json + Parquet→CSV only
python -m pipeline --inspect census_acs   # Print rows/columns of a processed source

# Serve the static site locally (opens at http://localhost:8000)
python -m http.server 8000

# Tests
pytest tests/ -v
pytest --cov=pipeline --cov=src --cov-report=term-missing
pytest tests/unit/test_lila_pipeline.py -v                 # single file
pytest tests/unit/test_lila_pipeline.py::test_name         # single test

# Lint / format
black . --line-length 88
flake8
```

## Architecture

```
project.yml  ──→  src/config_loader.py  ──→  pipeline/*  ──→  data/*
                                                                │
                     pipeline/export_config.py  ───────→  data/config.json
                     pipeline/export_csv.py     ───────→  data/*.csv
                                                                │
                                         fetch()  ←────────────┘
                                            │
                                   map.js (Leaflet) + {index,Map,About}.html
```

- **[project.yml](project.yml) is the single source of truth.** Layer definitions, indicator metadata (column, display label, palette, format string, caption, granularities, categorical flag), partner types (label/color/icon), palette color arrays, and geography (state FIPS, 14 MSA counties, map center, default zoom) all live here. `src/config_loader.py` is the YAML accessor.
- **`src/config.py` is env-driven only** — `APP_ENV`, `USE_MOCK_DATA`, `MOCK_DATA_DIR`, `AWS_BUCKET_NAME`, message constants. It has zero imports from other `src/` modules and does not depend on Streamlit.
- **Scripts in `scripts/` and modules in `pipeline/` must not import from `src/`** other than `src.config` and `src.config_loader` (both are Streamlit-free). Keep the boundary clean.
- **Custom exceptions live in the module that raises them** — `GeocodingError` in `pipeline/process_partners.py` and `pipeline/process_giving_matters.py`.
- **`data/` is gitignored.** It is regenerated by every pipeline run.

### The export step is load-bearing

`python -m pipeline` automatically runs `run_export_step()` at the tail, which calls `pipeline/export_config.py` (→ `data/config.json`) and `pipeline/export_csv.py` (mirrors every `data/*.parquet` to `data/*.csv`). **Without these, the frontend has nothing to fetch.** If you only re-ran a single step (e.g. `--step census_acs`), run `--step export` afterward to refresh `config.json` and CSVs.

The exporter renames `health_lila_*` → `health_*` on the CSV side so the frontend fetches `data/health_tract.csv` / `data/health_zip.csv`. Keep this rename — map.js hardcodes the short names.

### The config.json contract

`map.js` loads `data/config.json` at startup in `applyConfig()` and populates `PARTNER_TYPES`, `INDICATORS`, `PALETTES`, `PALETTE_META` from it. The shape is:

```json
{
  "project":       {"name", "slug", ...},
  "geography":     {"map_center", "default_zoom", "msa_counties_count"},
  "indicators":    [{id, label, col, src, granularities, palette, fmt, caption, categorical?}],
  "partner_types": {<type_id>: {label, color, icon}},
  "palettes":      {greens: [hex×5], oranges: [hex×5], ...},
  "palette_meta":  [{id, label}]
}
```

Each indicator's `fmt` is a Python-style format string (`${:,.0f}`, `{:.1f}%`, `{:,.0f}`, `{}`). `map.js::makeFormatter()` parses them. If you add a new format pattern to `project.yml`, extend the parser.

### Data contract (flat `data/`)

```
data/
├── config.json                 # emitted by pipeline/export_config.py
├── tracts.geojson              # 11-char GEOID (zero-padded), ~434 features
├── zipcodes.geojson            # ZCTAs (MultiLineString → MultiPolygon in-browser)
├── counties.geojson            # 14 MSA county polygons
├── msa.geojson                 # dissolved MSA boundary (single feature)
├── partners.geojson            # NFP partners — props: partner_name, address, partner_type
├── giving_matters.geojson      # CFMT orgs    — props: partner_name, address, partner_type (same keys!)
├── acs_tract.csv + .parquet    # DP03_0062E, DP03_0119PE, DP05_0001E
├── acs_zip.csv  + .parquet     # same columns
├── health_tract.csv + .parquet # DIABETES, BPHIGH, OBESITY
├── health_zip.csv  + .parquet  # same columns
└── usda_lila_tract.csv + .parquet  # LILATracts_1And10, lapop1, lalowi1 (TRACT ONLY)
```

Both point GeoJSONs use the same property keys (`partner_name`, `partner_type`) — this is deliberate. The original design mockup had `partners.geojson` using `partner_type` and `giving_matters.geojson` using `category`; we aligned on the former. If you edit either producer, keep them consistent.

## Invariants to preserve

1. **11-char zero-padded GEOIDs across every source.** `pipeline/load_source.py::normalize_geoid` does this; `pipeline/export_csv.py` re-applies the zfill to the CSV so string zeros survive round-tripping. ZCTA GEOIDs are technically 5 chars — the frontend (`parseCsv` in `map.js`) trims leading zeros back to 5.
2. **Geocoding happens in the pipeline, never at render time.** Per-address failures set lat/lon to NaN; the point feature is dropped in the GeoJSON writer. The pipeline logs a summary count.
3. **USDA LILA is tract-only.** The `tract_only: true` flag in `project.yml` causes `export_config.py` to emit `"granularities": ["tract"]`, which hides LILA indicators in the map's sidebar when the user picks ZIP.
4. **Giving Matters categories are LLM-assigned** via `scripts/classify_giving_matters.py` (Anthropic API). The `other` bucket is large by design — the CFMT directory is broader than NFP's partner types. The pipeline step gracefully skips when `enabled: false` or the S3 key is missing.
5. **Both point GeoJSONs use `partner_type`.** Do not reintroduce `category` or `name` keys.
6. **No Streamlit, no Folium, no Branca.** These were dropped when the Leaflet UI replaced Streamlit. Don't add them back to `requirements.txt` — the frontend is browser-side Leaflet.
7. **No build step.** Keep the HTML/CSS/JS hand-authored and directly servable via `python -m http.server`. Don't introduce npm, a bundler, TypeScript, or React without an explicit decision to change the deployment story.

## Mock mode vs. production

- `USE_MOCK_DATA=true` in `.env` → reads from `data/mock/*.csv`; geocode cache from `data/mock/mock_geocode_cache.csv` if present; no S3 writes. Run `scripts/generate_mock_data.py` + `scripts/generate_mock_parquet.py` once to populate mocks.
- `USE_MOCK_DATA=false` → boto3 reads from `s3://${AWS_BUCKET_NAME}/...`. Required env vars: `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_BUCKET_NAME`, and (for Giving Matters classification) `ANTHROPIC_API_KEY`.

## Deployment

Deployed on **Vercel** (project `nfp-food-insecurity-map`, team `databelmonts-projects`). The Vercel build runs `scripts/sync-data.mjs`, which downloads `s3://nfp-food-insecurity-map-data/current/` into local `data/` before serving the static HTML/JS. The Python pipeline is **not** run during the Vercel build — it runs out of band (locally or on a scheduled job) and the operator uploads its output to the output bucket with `aws s3 sync`.

**Two buckets, two roles — do not conflate:**
- `bdaic-public-transform` — SOURCE inputs. Referenced by `s3_bucket:` in [project.yml](project.yml) and `AWS_BUCKET_NAME` in `.env.example`. Pipeline reads only.
- `nfp-food-insecurity-map-data` — OUTPUT, served to the website, under the `current/` prefix. Pipeline writes; Vercel reads. The `current/` prefix leaves room for peer prefixes (`archive/`, `staging/`) added later. Hardcoded in `scripts/sync-data.mjs:10-11`.

**Deploy commands:**
```bash
# Code change
vercel --prod --scope databelmonts-projects

# Data change (pipeline re-run)
python -m pipeline
aws s3 sync data/ s3://nfp-food-insecurity-map-data/current/ --exclude "mock/*"
vercel --prod --scope databelmonts-projects     # Vercel does not auto-detect new S3 objects
```

GitHub auto-deploy is NOT wired — `vercel --prod` uploads the working tree directly. Commit first so git and prod match.

For IAM, env-var rotation, runbook entries, and the onboarding checklist, see [OPERATIONS.md](OPERATIONS.md).

## Gotchas

- **Don't edit `map.js` constants to add a new indicator or partner type.** They're regenerated from `project.yml` every pipeline run. Edit `project.yml`, run `python -m pipeline --step export`, refresh the browser.
- **Don't forget the rename.** If you add a new data source whose Parquet stem clashes with a frontend expectation (e.g. you write `health_lila_tract.parquet` but the frontend wants `health_tract.csv`), update `pipeline/export_csv.py::STEM_RENAMES`.
- **`index.html` computes its stats dynamically** from `data/config.json`, `data/tracts.geojson`, `data/partners.geojson`, `data/giving_matters.geojson`. If a file is missing, the counter shows `—`; don't hardcode numbers back in.
- **Serving via `file://` will break `fetch()`** for CORS reasons. Use `python -m http.server` (or any static server) to preview locally.
- **`src/geocoder.py` was deleted.** The pipeline's geocoding logic lives inline in `pipeline/process_partners.py` and `pipeline/process_giving_matters.py`. Don't resurrect `src/geocoder.py` — there's no Streamlit to cache into anymore.
- **A source-load failure does not abort the pipeline.** `pipeline/load_source.py::process_data_source` returns `None` on `FileNotFoundError`, and `pipeline/__main__.py::run_data_step` does not check the return value. A failing source will log an error and the run will still exit 0, potentially leaving a stale parquet on disk that gets re-uploaded. After any pipeline run against real S3, scan stderr for `No data found for ...` before running `aws s3 sync`. See [OPERATIONS.md §6](OPERATIONS.md) for the broader list of known gaps.
- **`python -m pipeline` is not a deploy.** It writes to local `data/`. To ship the result, you also need `aws s3 sync data/ s3://nfp-food-insecurity-map-data/current/` and then `vercel --prod`. Vercel does not re-pull from S3 on its own when new objects arrive.
